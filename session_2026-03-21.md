### 2. Bug Fix: `src/t_ragx/utils/elastic.py` — `upload_df()`
Three related bugs in the bulk indexing function:

**Bug 1**: `es_client.bulk()` called with positional arguments
elasticsearch-py 8.x only accepts keyword arguments.
`es_client.bulk(upsert_doc(...), index)` → `es_client.bulk(operations=upsert_doc(...), index=index)`

**Bug 2**: `index` not forwarded into `upsert_doc()`
`upsert_doc` hardcodes `"_index": "translation_memory"` in the action header unless the `index` parameter is passed. Since it wasn't, all documents were written to the default index `translation_memory` instead of the requested one (e.g. `zh_en_translation_memory`).
Fixed by passing `index=index` to `upsert_doc()`.

**Bug 3**: `except: raise r` swallowed the real exception
If the bulk call raised an exception, `r` was unbound, so `raise r` would itself raise `UnboundLocalError` instead of the real error — making all failures completely silent.
Fixed to `except Exception as e: raise e`.

These bugs caused TM retrieval to silently return no results for all sentences.

## Concepts Discussed

- **ES index**: equivalent of a database table; stores JSON documents and an inverted index for BM25 search
- **BM25 vs semantic search**: BM25 and Levenshtein are both lexical methods; semantic search uses embeddings for meaning-based similarity
- **Retrieval pipeline**: BM25 retrieves candidates → Levenshtein reranks → top-k returned
- **sim%, score, dist, normed_dist**:
  - `score`: BM25 relevance score from ES (used for initial retrieval only)
  - `dist`: raw Levenshtein character edit distance (used for reranking)
  - `normed_dist`: `dist / len(source_sentence)` (normalises for sentence length)
  - `sim%`: `(1 - normed_dist) * 100` (human-readable, derived from normed_dist alone)
- **normed_dist > 1.0**: possible when the retrieved segment is longer than the source; this contradicts the system prompt's claim that similarity is bounded at 1.0 — flagged as a known issue to fix later
